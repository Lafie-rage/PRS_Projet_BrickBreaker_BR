5 ## Introduction
6 
7 Ce repository contient l'un de nos projet de 3ème année en école d'ingénieur informatique et industriel.
8 
9 Le but était de créer un jeu multijoueur et le rendre aussi simple que possible afin que n'importe qui puisse le télécharger et y jouer sans soucis ou même continuer de le développer.
10 
11 **Attention** : ce jeu ne fonctionnera **uniquement sous système d'exploitation vasé sur Linux** pour l'instant. Il ne fonctionnera pas sous Windows et n'a pas été testé sous Mac OS.
12 Si vous souhaitez nous aider à le rendre cross-plateforme, n'hésitez pas. :)

14 ## Prérequis
15 
16 Il n'y a pas beaucoup de prerequis pour suivre ses instructions hormis savoir ouvrir une console linux (ctrl+alt+t sous Ubuntu) et savoir lire les info des commandes données.
17 Pour ce dernier, voici quelques rappels :
18 * **#** : signifie que la commande doit être exécuté entant que root.
19 * **$** : signifie que la commande peut être exécuté en utilisateur normal.
20 * **user:/path/to/folder $** : signifie que vous êtes l'utilsateur __user__ et que vous vous trouver dans le dossier dont le chemin est __/path/to/folder__. En général, ce n'est pas précisé, si ça l'est, c'est que c'est important d'être cet utilisateur dans ce répertoire.
21 Lorsque le répertoire n'est pas donné c'est que vous devez être dans le dossier du jeu.
22 * **~** : signifie le répertoire "home" de l'utilsateur. En général /home/<user> avec __<user>__ le nom de l'utilsateur.

71 
72 Afin de jouer seul il vous faudra simplement lancer un client et le serveur.
73 Pour ceci, vous devrez d'abord compiler chacun des executables. Pour ceci, ouvrez une console dans le répertoire du jeu et tapez-ceci :

124 ##### Connaitre l'IP du serveur
125 
126 Avant de vous connecter par ssh sur le serveur, il vous faudra connaître son IP.
127 Pour que tout fonctionne, il vous faudra être sur le même réseau (par exemple, sur la même box internet).
128 Pour connaître l'ip du serveur, utiliser la commande suivante sur votre pc serveur en remplacement __[interface]__ par l'interface réseau utilisé si vous la connaissez :
129 ```
130 $ ifconfig [interface]
131 ```
132 En général, votre ip sera du type 192.168.x.x (avec x un nombre entre 0 et 255).
133 Sous Ubuntu, vous pouvez aussi utiliser :
134 ```
135 $ hostname -I
136 ```
137 Cette commande vous donnera directement l'ensemble de vos adresses IP.
138 
139 Si vous avez plusieurs adresses, vous pouvez effectuer la même commande sur un pc client se trouvant sur le même réseau. Son seul le chiffre après le dernier '.' devrait avoir changé.
140 Par exemple:
141 IP serveur : 192.168.1.1
142 IP client : 192.168.1.2
143 
144 Si vous n'êtes toujours pas sûr, depuis un client essayer de ping les IP du serveur jusqu'à ce qu'un des ping vous donne une réponse. Vous pouvez faire ceci de cette manière (en remplace __<ipServer>__ par une des IP du serveur):
145 ```
146 $ ping <ipServer>
147 ```

211 ## Règles du jeu
212 
213 Le but du jeu est de **casser le mur de brique** (représentée par des *B*) à l'aide de la balle (représentée par *o*).
214 
215 Pour ceci, vous devrez **faire rebondir la balle** sur la raquette (représentée par une ligne de *~*), les murs ou les briques à détruire.
216 
217 Chaque fois que la balle touchera un obstacle, elle rebondira selon un angle à **45 degrés** dont la direction sera changé comme ceci:
218 * Depuis le haut gauche :
219 
220 ![bal_from_up_left](./ressources/bal_from_up_left.png)
221 
222 * Depuis le haut droite :
223 
224 ![bal_from_up_right](./ressources/bal_from_up_right.png)
225 
226 * Depuis le bas gauche :
227 
228 ![bal_from_down_left](./ressources/bal_from_down_left.png)
229 
230 * Depuis le bas droite :
231 
232 ![bal_from_down_right](./ressources/bal_from_down_right.png)
233 
234 Chaque brique cassée raporte **1 points**.
235 Dans une partie normale il y a **152 briques**.
236 
237 Chaque fois que la balle touche la partie sous la raquette, vous perdez **une vie**.
238 Vous disposez de **3 vies**.
239 Chaque vie perdue **divise vos points par deux en tronquant le résultat**.

241 ## Contrôle du jeu
242 
243 * *'q'* déplace la raquette à gauche.
244 * *'d'* déplace la raquette à droite.
245 * *'p'* interrompt ou reprend la partie.
246 * *'o'* accélère la partie. L'accélération n'est utile que lors que l'on active la raquette infinie afin de tester les maps. En effet, puisque nous ne faisont que des angles à 45° la plus part des configurations de map ne sont pas finissables.
247 * *'i'* active la raquette infinie. Comme dit précédemment, cette option est utilisé pour tester les patternes de maps.
248 * *'c'* arrête/abandonne la partie.

256 ### La grille du jeu
257 
258 En raison de l'utilisation de configuration de la tty en raw et sans écho (les raisons seront expliquée par la suite), nous avons des affichages parfois chaotique, le jeu est donc réglé pour être joué dans une console à taille par défaut sous Ubuntu (80x24).
259 
260 La grille de jeu est un tableau 2D de char faisant 80x20 et de deux lignes d'affichages d'informations sur la partie.
261 Le tableau est composé de différents char:
262 * *' '* pour les casses vides
263 * *'W'* pour les murs
264 * *'B'* pour les briques
265 * *'~'* pour la raquette
266 * *'o'* pour la balle
267 * *'R'* pour les erreurs de caractères
268 * *'E'* lors du replacement de la balle après une mort
269 
270 Chacun de ses caractères est affiché identiquement à l'écran à l'exception de *'E'* qui est remplacé par *' '* à l'écran puisque ce caractère sert à replacée la balle sur la raquette.
271 
272 L'intégralité des informations se trouvent dans des structures (ball_t, brick_t, ...).
273 Plus de détailles se trouvent dans la documentation créée à l'aide de doxygen.
274 
275 ### Affichage et récupération des saisis clavier
276 
277 La partie graphique du jeu repose sur un rafraichissement constant de l'affichage de la grille à l'écran sur un premier thread et un second thread qui attend la saisi d'un caractère par l'utilisateur pour réagir.
278 
279 Il y a plusieurs possibilités de réactions en fonction de la touche saisie:
280 * *'q'* déplace la raquette à gauche.
281 * *'d'* déplace la raquette à droite.
282 * *'p'* interrompt ou reprend la partie.
283 * *'o'* accélère la partie. L'accélération n'est utile que lors que l'on active la raquette infinie afin de tester les maps. En effet, puisque nous ne faisont que des angles à 45° la plus part des configurations de map ne sont pas finissables.
284 * *'i'* active la raquette infinie. Comme dit précédemment, cette option est utilisé pour tester les patternes de maps.
285 * *'c'* arrête/abandonne la partie.

287 ### Communication entre le serveur et le client
288 
289 #### Choix du protocole de communication
290 
291 Notre choix a été de prendre un protocole type TCP. C'est à dire qu'à chaque requête envoyé par le serveur (resp. le client), le client (resp. le serveur) envoit une réponse pour confirmer sa réception et son traitement.
292 
293 ### Diagramme de séquence
294 
295 En bleu, ce sont les communications par boîte aux lettres.
296 En rouge ce sont les communications par la mémoire partagée.
297 
298 ![Diagramme de séquence](./ressources/sequence_diagram.png)
299 
300 La connexion par SSH n'est pas obligatoire puisqu'on peut également jouer depuis le PC serveur.
301 
302 La séquence de communication type est donc la suivante :
303 1. A son lancement, le serveur attent les messages des clients sur la boîte aux lettres. Il sait combien de clients doivent se connecter puisque cette information lui est envoyé en paramètre.
304 1. A son lancement, le client envoit un message comportant le nom du joueur et son PID. C'est données sont stockée dans une structure contenant la liste des clients avec leur PID, le nom du joueur et une place pour son futur score.
305 1. Lorsque tous les clients attendus se sont connectés au serveur, ce dernier envoit un message à chaque client pour lui dire de lancer la partie.
306 1. Le client envoit un message pour confirmer que la partie s'est lancée puis lance la partie.
307 1. Lorsque la partie d'un client est terminé, un message est envoyé via la boîte aux lettres au serveur. Ce message contient le score des joueurs.
308 1. Le serveur attent la réception de chaque score et les stock dans la liste des clients.
309 1. Le serveur écrit le score dans la mémoire partagée.
310 1. Le serveur envoit un message à chaque client via la boite aux lettres pour leur dire que les scores sont écrit.
311 1. Les clients lisent les scores 1 à 1. Les accès concurents sont gérés par un sémpahore.
312 1. Le client répond au serveur lorsqu'il a lu le score puis l'affiche à l'écran.

314 ### Structure des fichiers sources
315 
316 Le code est répartit entre différents fichiers.
317 Il y a une librairie global au client et au serveur : kassbriik.h/.c.
318 Les executables et les libraires qu'ils utilisent sont :
319 * client : game_ui.h/.c.
320 * server : server_kassbriik.h/.c.
321 
322 Des scripts bash servant à la configuration sont également disponibles :
323 * creating_players_accounts.sh : Création et configuration du compte "player" qui est restreint (utilisation du rbash).
324 * move_client.sh : Déplace l'executable dans le dossier /bin/ afin d'être appelé comme une commande. Il ajoute le r au mode du fichier afin que le bit SUID soit utilisable depuis le rbash. Ce fichier est executé par défaut à la compilation du client à l'aide du makefile.
325 * set_up_config_server_ssh.sh : Remplace le fichier de configuration de sshd par la configuration du jeu. Stock l'ancienne configuration(si elle existe) dans le dossier "OLD_SSH_SERVER_CONFIG" à la racine du projet. Ce dossier disposera de différents dossiers "versionX".
326 
327 Le dossier offre aussi un fichier makefile permettant la compilation du projet par différents cibles. L'ensemble des executable compilée se trouvent dans le dossier build qui est créé lors de l'appel de n'importe quel cible hormis clean.
